## Ответы:

### Задание 1. СУБД

1.1 Рекомендуемая СУБД - реляционная (например, PostgreSQL, Microsoft SQL Server).

Обоснование:

Реляционные СУБД поддерживают ACID‑транзакции (атомарность, согласованность, изоляция, долговечность), что критично для финансовых данных.
Такая СУБД имеет четкую структуру, а табличная модель с предопределёнными схемами идеально подходит для структурированных финансовых данных (бюджеты, статьи расходов, отчёты).
Аналитические возможности этого типа СУБД подходят для сложных запросов, генерации отчётов.

1.1 Ускорение хеширования*

Для ускорения хеширования можно использовать:

Встроенные функции PostgreSQL (например, digest() из расширения pgcrypto).
Специализированные библиотеки на стороне приложения. Например:
для Python — hashlib;
для Node.js — встроенный модуль crypto.

1.2. Для лендингов: документо‑ориентированная NoSQL‑СУБД (например, MongoDB).

Обоснование:

Это гибкая СУБД, для случаев когда структура данных с лендингов может меняться, такая СУБД позволяет хранить данные без жесткой схемы.
Она имеет быструю запись данных с множества лендингов и её можно горизонтально масштабировать в ависимости от ситуации.

Для CRM - реляционная СУБД (например, PostgreSQL, MySQL).

Обоснование:

Необходимость рабортать с множеством связанных сущностей (клиент - сделка - менеджер), это эффективно в реляционной БД. Также есть возможность эффективно работать с аналитикой.

1.2 Да, можно использовать например PostgreSQL.

Использовать реляционные таблицы для сущностей CRM, и JSON-поля для данных с лендингов.

1.3. Рекомендуемая СУБД - документо‑ориентированная NoSQL (например, MongoDB).

Обоснование:

Данные структурированы (по отделам, должностям). СУБД достаточно гибкая для различной структуры и поддерживает полнотекстовый поиск.

1.3 Можно использовать PostgreSQL из задачи 1.2.

В JSON-полях хранить статьи, инструкции, метаданные. Реляционные таблицы использовать для сваязи должностей и отделов. Также PostgreSQL поддерживает полнотекстовый поиск.

1.4. Можно реализовать с использованием графовой СУБД (например, Neo4j, Amazon Neptune).

Обоснование:

Сама по себе это графовая задача, с узлами (объектами, курьерами) и ребрами (маршрутами). Так что мы получим легкое добавление и изменение узлов и маршрутов, скорость работы (т.к. Графовые СУБД оптимизированы для обхода связей и поиска путей).

1.4 Лучше использовать одну графовую СУБД. 

Мы можем добавить узлы для отдела закупок и дополнительные связи, и получим один источник данных для логистики и закупок, возможность строить сквозные отчеты, и возможность оптимизировать маршруты с учетом закупок (например поставка нескольким поставщикам одним курьером).

1.5* Теоретически да, практически — с оговорками. 

Лучший кандидат — PostgreSQL.

Как это реализовать:

Мы можем использовать реляционные таблицы для Бюджетирования проектов, для лендингов - JSON и реляционные таблицы, для базы знаний - JSON, таблицы и полнотекстовый поиск, а для Логистики использовать расширение, например Apache AGE.

Однако наряду с плюсами: одной точной управления данными, одним языком запросов мы получим и минусы: сложность настройки и администрирования, возможность потери производительности.

Также есть вариант мультимодельных СУБД, например ArangoDB, OrientDB. Однако они менее распространены.

### Задание 3. SQL vs NoSQL

3.1. Пять преимуществ SQL‑систем по отношению к NoSQL

1) Гарантированная целостность данных (поддержка ACID - атомарность, согласованность, изоляция, долговечность). Это критично для финансовых операций, учёта и других сценариев, где недопустимы расхождения или потеря данных.
2) Чёткая и жёсткая схема данных. В SQL структура данных определяется заранее (таблицы, колонки, типы данных, ограничения)
3) Мощный и стандартизированный язык запросов (SQL)
4) Развитая экосистема инструментов. Например: BI‑системы (Power BI, Tableau); ORM‑библиотеки (Hibernate, Entity Framework, SQLAlchemy) средства мониторинга и администрирования.
5) Надёжность СУБД. SQL‑системы существуют десятилетиями, их архитектура хорошо изучена. Они прошли проверку в самых требовательных средах (банки, госучреждения, ERP‑системы).

### Задание 4. Кластеры

Ключевой критерий — характер нагрузки и тип данных. А оптимальная модель будет зависит от типа вычислений.
Например:

Аналитические нагрузки:

Тип данных: большие объёмы структурированных/полуструктурированных данных.
Требования: высокая пропускная способность для чтения, поддержка сложных аналитических запросов.
Подходящие СУБД: колоночные распределённые (ClickHouse, Apache Druid), а также решения на базе Hadoop/Hive.

Потоковая обработка:

Тип данных: непрерывный поток событий (кликстримы, IoT‑сенсоры, трейдинг).
Требования: обработка с минимальной задержкой (миллисекунды), устойчивость к сбоям.
Подходящие СУБД: Apache Kafka + потоковые движки (Apache Flink, Spark Streaming), либо специализированные потоковые БД (Apache Druid, ksqlDB).
